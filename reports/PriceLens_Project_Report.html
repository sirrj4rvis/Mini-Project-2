PriceLens Project Report02

Abstract

01

PriceLens is an intelligent e-commerce price comparison platform developed as a three-service application. The frontend provides a modern product discovery interface, search results, product detail pages, analytics, price alerts, a dashboard, admin views, and community discussion pages. The backend exposes a REST API for authentication, product search, comparison, alerts, admin analytics, forum posts, and communication with the ML service. The ML service provides price prediction and recommendation endpoints and also serves as the bridge for Flipkart scraping.

The implemented system aggregates product data from Amazon through RapidAPI, eBay through the official Browse API, Walmart through RapidAPI, and Flipkart through a Python scraping endpoint. Products are normalized into a shared format, stored in MongoDB, and enriched with price history records. The platform then uses these records to show comparisons, charts, and machine-learning-assisted recommendations.

Introduction

02

Online shoppers often compare prices manually across multiple websites before making a purchase. This process is time-consuming and can still miss better deals, historical price patterns, or future price drops. PriceLens addresses this problem by combining product aggregation, price comparison, alerting, historical tracking, and prediction in one application.

Problem Statement

The project aims to design and implement a full-stack platform where users can search for a product, compare its prices across available marketplaces, inspect price history, and receive actionable recommendations about whether to buy now or wait.

Need for the System

  • Prices vary across platforms and can change frequently.
  • Manual comparison does not preserve historical pricing trends.
  • Users need alerts when a product reaches a target price.
  • Decision support is more useful when historical data and prediction are combined.

Implemented Outcome

The delivered project includes a React frontend, Express backend, MongoDB database layer, and Flask ML microservice. It supports product search, marketplace comparison, product detail pages, watchlist behavior, price alert creation, admin statistics, forum discussions, and ML-based recommendation fallbacks.

System Architecture

05

PriceLens follows a modular architecture where the frontend communicates with the Express backend, the backend persists data in MongoDB, and ML-related requests are delegated to the Flask microservice. External product integrations are isolated in service files and normalized before being stored.

High-Level Request Flow

  1. User searches for a product from the frontend search page.
  2. The frontend calls GET /api/products/search?q=....
  3. The backend queries Amazon, eBay, Flipkart, and Walmart services where available.
  4. Raw results are normalized into a common product format.
  5. Products are upserted into MongoDB and price history points are inserted.
  6. The frontend receives unified product responses and renders comparison cards.
  7. Product detail pages call prediction and history APIs for deeper decision support.

Backend Implementation

06B

The backend is an Express REST API. It applies Helmet, CORS, JSON parsing, global rate limiting, request logging, route-level validation, centralized error handling, and MongoDB persistence through Mongoose.

Controller Responsibilities

ControllerResponsibility
authControllerRegistration, login, refresh token, profile, logout.
productControllerSearch, upsert, product details, history, compare, prediction, redirect, trending, autocomplete, watchlist, reviews.
alertControllerCreate, list, update, pause/resume, delete, and test alerts.
adminControllerDashboard stats, users, products, alerts, and admin management operations.
forumControllerForum posts, comments, votes, user posts, and protected edits.

Product Search Pipeline

  1. Validate query parameters using express-validator.
  2. Build or read a cache key for repeated searches.
  3. Fetch marketplace data in parallel with Promise.allSettled.
  4. Normalize Amazon, eBay, Flipkart, and Walmart results.
  5. Merge products by normalized title.
  6. Upsert products and insert price history entries.
  7. Apply category, price, and platform filters.
  8. Sort and return a unified response.

Security and Reliability

  • JWT authentication protects private routes such as alerts, dashboard data, watchlist, and admin APIs.
  • Passwords are hashed with bcrypt before saving.
  • Helmet, CORS, validation, and rate limiting are configured in the Express app.
  • ML calls use retry behavior, caching, and fallbacks when the Flask service is unavailable.

Normalization

The productNormalizer service converts platform-specific responses into one product shape with fields for title, brand, image, category, description, ASIN, search keywords, and source-specific price data. Currency conversion normalizes source prices into the configured default currency, which defaults to INR.

ML Bridge

The Express backend does not directly run model inference. Instead, it sends product history and metadata to the Flask service through mlBridgeService. This bridge adds retries, timeout handling, cache support, payload normalization, and graceful fallback responses.

Feature Modules

06D

User Authentication

Users can register, log in, refresh tokens, fetch their profile, update profile data, and log out. Private frontend requests use access tokens stored in local storage and automatically request refresh tokens when a private request returns 401.

Price Alerts

Alerts are protected by JWT authentication. Each alert stores userId, productId, targetPrice, optional percentage drop target, baselinePrice, platform, active status, triggered state, notification flags, cooldown hours, recurring behavior, and notes.

Watchlist and Dashboard

Users can add or remove products from their watchlist through the product watchlist endpoint. The dashboard fetches watchlist product data and user alerts to show tracked products, basic savings, alert counts, and category summaries.

Admin Module

Admin UI guards access by checking user role. The admin area contains overview statistics, user management, and product/alert views. Some chart series in the UI are presentation placeholders because backend time-series analytics are not fully implemented.

Forum Module

Forum APIs allow listing posts publicly, reading individual posts, filtering by tag and query, creating posts with authentication, editing or deleting owned posts, adding comments, editing comments, deleting comments, and voting on posts or comments.

Product Source Subdocument

Each Product contains an embedded sources array. Every source stores platform, price, currency, original price, discount, link, availability, rating, review count, seller, and last updated time. This allows one product record to hold marketplace-specific current prices without requiring a separate collection for current source snapshots.

Indexes

  • Product text index over title, searchKeywords, and brand.
  • Product indexes over lowestPrice, category plus lowestPrice, and searchCount.
  • Alert indexes for active alert cron checks, user alert dashboard queries, and product-level alert lookups.
  • User indexes for role and active status plus recent creation sorting.

Machine Learning Implementation

09

The ML service is implemented as a separate Flask application. It loads serialized model artifacts from the models directory when available and exposes prediction, recommendation, batch prediction, model report, health, and Flipkart scrape endpoints.

Prediction Logic

  1. Receive product price history and optional metadata from Express.
  2. Require at least three historical price points for prediction.
  3. Build date, lag, rolling window, momentum, category, platform, rating, review count, and MRP features.
  4. Use trained pipeline artifacts when present.
  5. Blend primary model output with linear regression when that artifact is available.
  6. Fallback to linear extrapolation when trained artifacts are unavailable.
  7. Return predicted price, confidence, price range, method, and current price.

Recommendation Logic

A deterministic rule-based recommendation converts predicted movement into BUY_NOW, WAIT, or WATCH. A falling trend with meaningful savings recommends waiting. A rising or stable trend tends toward buying now. Unknown or insufficient data produces watch-style responses.

Limitations and Future Scope

10B

Current Limitations

  • External marketplace coverage is limited to the integrations present in code.
  • External APIs require valid credentials and may return empty results when keys are missing or rate limits are reached.
  • ML accuracy depends on sufficient historical price records and available trained artifacts.
  • Homepage products are demo data, not database records.
  • Some admin and dashboard visualizations are UI placeholders rather than fully backend-driven time-series analytics.
  • Mock product purchase URLs search marketplaces instead of opening exact listings.

Future Enhancements

  • Add more official marketplace APIs with stable product identifiers.
  • Replace homepage demo products with backend-driven trending products.
  • Add full frontend automated tests with component and route coverage.
  • Add backend time-series analytics for admin activity charts.
  • Improve model training pipeline documentation and automate periodic retraining.
  • Add a browser-safe redirect tracking endpoint for all external purchase clicks.
  • Improve alert UX for percentage-drop alerts and recurring alert controls.

Deployment Considerations

Production deployment should define environment variables for MongoDB, JWT secrets, API keys, email credentials, client origin, and ML service URL. The frontend should point VITE_API_URL to the deployed backend. Backend and ML service logs should be monitored because marketplace integrations depend on external services.

Conclusion

11

PriceLens successfully implements a practical full-stack product comparison and price intelligence platform. It combines a modern React frontend, Express API backend, MongoDB persistence layer, and Python Flask ML service into a working architecture. The project supports product search, marketplace comparison, product detail pages, price history, alerts, dashboards, admin management, community discussions, and recommendation logic.

The corrected implementation shows that the strongest part of the project is its modular service design: marketplace adapters, product normalization, MongoDB persistence, and ML bridging are separated cleanly enough to extend later. The system is not a universal marketplace monitor, but it is a functional and extensible foundation for intelligent e-commerce comparison.
